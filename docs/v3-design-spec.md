## 1. 모듈 구조 ✅ 확정

### 1-1. 모듈 정의

과제의 기본 단위는 **모듈**이며, 하나의 모듈 안에 여러 유형의 문제가 포함됩니다.

| 섹션 | 모듈 구성 | 총 문제 수 | 시간 제한 |
|---|---|---|---|
| **Reading** | 빈칸채우기×2 + Daily1×2 + Daily2×2 + Academic×1 | 35문제 | 20분 |
| **Listening** | 응답고르기 + 대화 + 공지사항 + 렉쳐 | 전 유형 포함 | 유형별 제한 |
| **Writing** | Arrange + Email + Discussion | 전 유형 포함 | 유형별 제한 |
| **Speaking** | Repeat×1 + Interview×1 | 11문제 | 유형별 내부 타이머 |

### 1-2. 핵심 규칙

- 모듈은 **전체를 한 번에** 풀어야 함 (특정 유형만 골라 풀기 불가)
- 오답노트 = **모듈당 1개** (예: 리딩 모듈 1에 대한 오답노트 1개)
- 채점 대시보드 = 모듈 내 **모든 유형의 결과를 유형별로** 표시
- 모듈 번호에 따라 세트 ID가 자동 증가 (module-definitions.js 규칙 유지)

## 2. 과제 화면 구조 ✅ 확정

### 2-1. 과제 진행 순서

```
과제 버튼 클릭 → TaskDashboard
  → [실전풀이] 활성화 + [해설보기] 잠김
  → 실전풀이 완료 → initial_record 저장
  → [해설보기(+메모장)] 잠금 해제, [실전풀이] → "다시풀기"로 변경
  → 이후: [다시풀기] + [해설보기] 자유롭게 사용 가능
```

### 2-2. 실전풀이

- 기존 Component를 그대로 실행하여 문제를 품
- 풀이 완료 → initial_record로 저장 (최초 1회, 이후 변경 불가)

### 2-3. 다시풀기

- 2차 풀이 로직 **완전 폐지**
- 다시풀기 = 실전풀이와 **완전히 동일한 화면**을 다시 실행
- 결과는 current_record 컬럼에 **덮어쓰기** (매번 최신 결과로 대체)
- **current_record = initial_record와 완전히 동일한 JSON 구조** (같은 필드, 같은 형식)
- **저장 시점**: 다시풀기 채점이 끝난 직후 자동 저장 (실전풀이의 initial_record 저장과 동일한 시점) ✅ 확정
- 학생에게 보이는 화면은 실전풀이와 완전히 동일
- 횟수 제한 없음
- **마감 후에도 다시풀기 가능** — 단, 인증률에는 영향 없음 (current_record만 변경, initial_record 불변)

#### 2-3-1. 라이팅 다시풀기 특수 처리

- **Arrange**: 채점 결과가 current_record에 저장 (일반 다시풀기와 동일)
- **Email / Discussion**: 다시 작성 후 Submit 시 **자동으로 .txt 파일 다운로드 + current_record에 작성 내용 포함 저장** ✅ 변경 (2026-03-05)
  - writing_email_text, writing_discussion_text **별도 컬럼**은 **최초 1회만 저장, 이후 변경 불가** (initial_record와 동일 원칙)
  - 다시풀기 작성 내용은 **current_record JSON 안에 포함**하여 저장 (해설보기에서 "내가 쓴 글" 표시를 위해 필요)
  - TXT 다운로드도 동시 수행 — 다운로드 파일 내용: 문제 정보(Scenario, Task, Instructions) + 학생 작성 내용

#### 2-3-2. 스피킹 다시풀기

- 다시풀기에서 스피킹 녹음 파일을 **별도로 제출하지 않음**
- current_record에 완료 여부만 기록

### 2-4. taskDashboardScreen 버튼 구조 ✅ 확정

**V3에서는 버튼 2개만 사용합니다:**

```
┌─────────────────────────┐
│ 풀이                      │
│  [📝 실전 풀이]             │  ← 풀이 전 (풀이 후 "🔄 다시 풀기"로 변경)
│                          │
│ 해설                      │
│  [📖 해설 보기]             │  ← 메모장(오답노트) 포함
└─────────────────────────┘
```

- V2의 "2차 풀이" 버튼 → **제거** (다시풀기로 대체)
- V2의 "오답노트 보기" 버튼 → **제거** (해설보기 안 메모장으로 통합)

### 2-4-1. 버튼 상태 변화

| 상태 | 풀이 버튼 | 해설보기 버튼 | 대시보드 |
|---|---|---|---|
| 아직 안 품 (마감 전) | **📝 실전 풀이** (활성화) | 잠김 (비활성화) | 빈 상태 |
| 실전풀이 완료 (마감 전) | **🔄 다시 풀기** (활성화) | **📖 해설 보기** (활성화) | 점수 표시 |
| 마감 지남 + initial_record 없음 | **🔄 다시 풀기** (활성화) | 잠김 (비활성화) | 빈 상태 + 마감 배너 |
| 마감 지남 + initial_record 있음 | **🔄 다시 풀기** (활성화) | **📖 해설 보기** (활성화) | 점수 표시 + 마감 배너 |
| 마감 지남 + current_record만 있음 | **🔄 다시 풀기** (활성화) | **📖 해설 보기** (활성화, 다시풀기 탭만) | current_record 점수 + 마감 배너 |

### 2-4-2. 마감 안내 배너

마감이 지난 과제의 taskDashboardScreen에 표시:
> ⏰ 이 과제의 마감이 지났습니다. 다시 풀기로 연습할 수 있지만, 인증률에는 반영되지 않습니다.

### 2-5. 해설보기

- **기간 제한 없이** 언제든 자유롭게 재열람 가능 (열고 닫기 자유)
- 접근 경로: 스케줄 → 과제 버튼 → taskDashboardScreen → 해설보기 버튼
- **해설보기를 열려면 initial_record 또는 current_record 중 하나 이상이 존재해야 함** (둘 다 없으면 잠김)
- 재접속 시 DB의 initial_record / current_record에서 데이터를 불러와 해설 화면을 복원

#### 2-5-0. 해설 탭 구조 (Initial / Current 분리) ✅ 확정

해설보기 좌측에 탭 2개로 구분:

```
┌──────────────────┬──────────────────┐
│ 📋 실전풀이 (29/35) │ 🔄 다시풀기 (33/35) │
└──────────────────┴──────────────────┘
```

| 탭 | 데이터 소스 | 표시 조건 |
|---|---|---|
| **실전풀이** | `initial_record` | 항상 표시 (풀이 완료 시 존재) |
| **다시풀기** | `current_record` | 다시풀기를 한 적이 있을 때만 표시 |

- 두 탭은 선택한 답이 다르므로 **해설 내용도 각각 다르게 표시**
- "다시풀기" 탭은 **가장 최근 결과만** 표시 (current_record는 덮어쓰기이므로 항상 최신 1건)
- **우측 메모장은 탭과 무관하게 항상 동일** (오답노트는 과제당 1개)

- 해설 화면 = 기존 Result 화면을 좌측에 표시
  - **라이팅(Email/Discussion)**: Result 화면에 "내가 작성한 글" + "모범 답안" 이미 포함됨 → 메모장에 별도 표시 불필요
- 우측에 **메모장** (스플릿 스크린)
  - 메모장은 **탭에 따라 해당 오답노트를 표시** ✅ 확정
  - 실전풀이 탭 → `error_note_text` (initial_record용)
  - 다시풀기 탭 → `current_error_note_text` (current_record용)

#### 2-5-A. 메모장 탭별 동작 ✅ 확정

**실전풀이 탭의 메모장 (`error_note_text`):**

| 상태 | 조건 | 메모장 표시 |
|---|---|---|
| **작성 가능** | initial_record 있음 + 오답노트 미제출 + 마감 전 | 빈 메모장, 자동 저장, 제출 버튼 활성화 |
| **읽기전용** | 오답노트 이미 제출됨 | 제출된 내용 표시, 수정 불가 |
| **잠김** | 마감 후 + 오답노트 미제출 | "마감으로 인해 작성이 불가합니다" 안내 |

- 제출 시 → `error_note_text` DB 저장 + `error_note_submitted` = true + **인증률 반영**
- 제출 후 수정 불가

**다시풀기 탭의 메모장 (`current_error_note_text`):**

| 상태 | 조건 | 메모장 표시 |
|---|---|---|
| **작성 가능** | current_record 있음 | 메모장 표시, 자동 저장, 제출 버튼 활성화 |
| **읽기전용** | 이전에 작성한 내용이 있음 | 기존 내용 표시, 다시 풀기 후 새로 덮어쓰기 가능 |

- 제출 시 → `current_error_note_text` DB 저장 (덮어쓰기)
- **인증률에 영향 없음**
- 다시 풀기를 할 때마다 새로 작성 가능 (매번 덮어쓰기)

#### 2-5-B. 메모장 상세 동작

  - **자동 저장** (작성 가능 상태만): 마지막 입력 후 2초 뒤 localStorage에 자동 저장 + "자동 저장되었습니다" 토스트 표시
  - 새로고침/재진입 시 localStorage에서 내용 복원
  - 하단에 **최종 제출** 버튼 (20단어 이상 시 활성화, 스페이스 기준 카운트, 영어·한국어 무관)
  - 최종 제출 시 → DB 저장 + localStorage 삭제
  - 메모장은 **모듈당, 탭당 각 1개** (실전풀이용 1개 + 다시풀기용 1개)

#### 2-5-C. 오답노트와 initial_record / current_record의 관계 ✅ 확정

| 구분 | 컬럼 | 인증률 반영 | 제출 규칙 |
|---|---|---|---|
| **실전풀이용** | `error_note_text` + `error_note_submitted` | ✅ 반영 (50% → 100%) | 1회 제출 후 영구 잠금 |
| **다시풀기용** | `current_error_note_text` | ❌ 무관 | 매번 덮어쓰기 가능 |

- current_record는 인증률에 일절 관여하지 않음 (풀이든 오답노트든 무관)
- 인증률 100%의 "오답노트 제출"은 **반드시 initial_record에 연결된 오답노트만** 해당

#### 2-5-1. 스피킹 과제 오답노트 특수 처리

- 메모장 하단에 **녹음 파일 첨부** UI (파일 선택 버튼, 라벨: "스피킹 파일")
- 학생이 휴대폰으로 녹음한 파일을 선택하여 업로드하는 방식
- Supabase Storage 버킷 `speaking-files`에 업로드 (기존 코드 재사용)
- 파일 미첨부 시 제출 버튼 비활성화

#### 2-5-2. speaking_file 분리 원칙

| 컬럼 | 용도 | initial_record 시점 | current_record 시점 |
|---|---|---|---|
| `speaking_file_1` | 오답노트 전용 (실전풀이용만) | 최초 제출 시 저장, 이후 변경 불가 | 건드리지 않음 |
| `initial_record` | 실전풀이 결과 | 최초 저장, 불변 | 건드리지 않음 |
| `current_record` | 다시풀기 결과 | - | 덮어쓰기 (녹음 파일 경로 미포함) |

#### 2-5-3. Current Record 스피킹/녹음 안내 ✅ 확정

- 다시풀기 탭의 스피킹 오답노트에서는 **녹음 파일 첨부 불필요**
- 파일 첨부 UI를 표시하지 않음 (실전풀이 탭 전용)
- 대신 안내 메시지 표시: **"다시풀기 오답노트의 녹음 파일은 저장되지 않습니다. 개인 연습용으로만 활용해 주세요."**

### 2-6. 채점 대시보드 (taskDashboardScreen 우측) ✅ 확정

모듈 내 모든 유형의 결과를 유형별로 표시합니다.

**Reading 예시:**
```
[채점 결과 - 실전풀이]
├── 빈칸채우기 Set 1: 8/10 정답
├── 빈칸채우기 Set 2: 9/10 정답
├── Daily 1 Set 1: 2/2 정답
├── Daily 1 Set 2: 1/2 정답
├── Daily 2 Set 1: 3/3 정답
├── Daily 2 Set 2: 2/3 정답
├── Academic: 4/5 정답
├─────────────────────────
├── 총점: 29/35 (Level 3.0)

[최근 다시풀기] (있을 때만 표시)
├── ... (동일 형식)
```

**Listening 예시:**
```
[채점 결과 - 실전풀이]
├── 응답고르기: X/Y 정답
├── 대화: X/Y 정답
├── 공지사항: X/Y 정답
├── 렉쳐: X/Y 정답
├─────────────────────────
├── 총점: XX/YY (Level X.X)
```

**Writing 예시:**
```
[채점 결과 - 실전풀이]
├── Arrange: 8/10 정답
├── Email: 작성 완료 (108 words)
├── Discussion: 작성 완료 (95 words)
```

**Speaking 예시:**
```
[채점 결과 - 실전풀이]
├── 따라말하기: 완료 ✅
├── 인터뷰: 완료 ✅
```

- initial_record 기반 결과: 항상 표시
- current_record 기반 결과: 다시풀기 기록이 있을 때만 표시

### 2-7. 체크리스트 (스케줄 화면 과제 버튼) ✅ 확정

- 과제 버튼 옆에 ✅ 체크 아이콘 표시 (initial_record 존재 시)
- **MVP에서는 ✅ 단일 상태** (풀이 완료 여부만, 오답노트 구분 없음)
- 이후 피드백에 따라 2단계 구분(50%/100%) 추가 검토
- "X개 중 Y개 완료" 등의 진도 요약 카드는 **MVP에서 제외**
- 과제 완료 판단 = `initial_record`가 존재하는지 여부
  - 모든 문제를 완료하고 채점이 끝난 후에만 initial_record가 생성됨
  - 중간 이탈 시에는 initial_record 미생성 → 과제 미완료 상태 유지

## 3. 기록 구조 ✅ 확정

### 3-1. 데이터 저장 구조

```
과제 레코드 (1행)
├── initial_record 컬럼  ← 최초 실전풀이 결과. 절대 변경 불가. 박제.
│   └── 인증률, 주간 포트폴리오, 점수 추이 차트에 사용
│
├── current_record 컬럼  ← 다시풀기 결과. 매번 덮어쓰기. initial_record와 동일 구조.
│   └── 채점 대시보드에서 "최근 다시풀기"로 표시
│
├── writing_email_text    ← Email 작성 원문. 최초 1회만 저장. 불변.
├── writing_discussion_text ← Discussion 작성 원문. 최초 1회만 저장. 불변.
│
├── error_note_text       ← initial_record용 오답노트 (제출 후 수정 불가, 인증률 반영)
├── error_note_submitted  ← initial_record용 오답노트 제출 여부
│
├── current_error_note_text ← current_record용 오답노트 (덤어쓰기, 인증률 무관)
│
└── speaking_file_1       ← 스피킹 녹음 파일 경로 (오답노트 전용, 다시풀기와 무관)
```

### 3-2. DB 테이블: study_results_v3 (신규 생성) ✅ 확정

V2 테이블에 데이터 0건 (학생 미사용) → V3 전용 테이블을 처음부터 새로 생성.

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | UUID (PK) | 자동 생성 |
| `user_id` | TEXT | 사용자 ID |

## 7. 인증률 ✅ 확정

### 7-1. 인증 기준 (과제별)

**메인 4섹션 (reading, listening, writing, speaking):**

| 단계 | 조건 | 인증률 |
|---|---|---|
| 미완료 | 아직 안 풀 | **0%** |
| 실전풀이 완료 | initial_record 존재 | **50%** |
| 실전풀이 + 오답노트 제출 | initial_record + error_note_submitted | **100%** |

**단어 테스트 (vocab):**

| 단계 | 조건 | 인증률 |
|---|---|---|
| 미완료 / 정답률 30% 미만 | - | **0%** |
| 완료 + 정답률 30% 이상 | initial_record 존재 + accuracy ≥ 30 | **100%** |

**입문서 정독 (intro-book):**

| 단계 | 조건 | 인증률 |
|---|---|---|
| 미제출 / 20단어 미만 | - | **0%** |
| 20단어 이상 작성 후 제출 | initial_record 존재 | **100%** |

→ 메인 4섹션은 실전풀이(50%) + 오답노트(100%) 2단계.
→ 단어 테스트 / 입문서는 오답노트가 없으므로 조건 충족 시 바로 100%.

### 7-2. 계산

```
요일 인증률 = 해당 요일 각 과제 인증률 합계 / 해당 요일 전체 과제 수
주간 인증률 = 해당 주 각 과제 인증률 합계 / 해당 주 전체 과제 수
```

※ 메인 4섹션은 과제당 0% / 50% / 100% 중 하나, 단어·입문서는 0% / 100% 중 하나

### 7-3. 마감과 인증률 ✅ 확정

#### 마감의 의미
- 마감 시점에 **initial_record 전체가 잠김** (풀이 결과 + 오답노트 포함)
- `locked_auth_rate` 컬럼에 현재 인증률 확정 저장 → 이후 변경 불가
- 마감 후에는 **모든 활동이 current_record에만 기록됨**

#### 시나리오별 마감 처리

**시나리오 A: 마감 전에 실전풀이 + 오답노트까지 완료한 학생**

| 시점 | 상태 |
|---|---|
| 마감 전 | initial_record ✅, error_note_submitted ✅ |
| 마감 시 | locked_auth_rate = **100%** 확정 |
| 마감 후 | 다시풀기(current_record) 가능, 해설보기(양쪽 탭) 가능 |

**시나리오 B: 마감 전에 실전풀이만 하고, 오답노트를 못 쓴 학생**

| 시점 | 상태 |
|---|---|
| 마감 전 | initial_record ✅, error_note_submitted ❌ |
| 마감 시 | locked_auth_rate = **50%** 확정 |
| 마감 후 | initial_record 잠김 → 오답노트 제출 불가 |
| 마감 후 | 다시풀기(current_record) 가능, 해설보기(양쪽 탭) 가능 |
| 오답노트를 쓰고 싶다면 | current_record 풀이를 먼저 완료 → current_record 해설에서 오답노트 작성 |

**시나리오 C: 마감 전에 아무것도 못 한 학생**

| 시점 | 상태 |
|---|---|
| 마감 전 | initial_record ❌ |
| 마감 시 | locked_auth_rate = **0%** 확정 (행이 없으면 0으로 간주) |
| 마감 후 | initial_record는 영구히 빈 상태로 잠김 |
| 마감 후 풀기 시작 | **current_record에 저장** (initial이 아님) |
| 해설보기 | current_record가 생긴 후에만 가능 (다시풀기 탭만 표시) |
| 대시보드 | initial_record 영역 = 빈 상태, current_record 영역 = 점수 표시 |

#### locked_auth_rate 동작 원리

| 상태 | 인증률 산출 방식 |
|---|---|
| `locked_auth_rate` = NULL | 마감 전 → initial_record + error_note 상태로 **실시간 계산** |
| `locked_auth_rate` = 값 있음 | 마감 후 → 해당 값 **고정 사용** (더 이상 변하지 않음) |

잠금 시점: 학생이 마감 지난 과제의 taskDashboardScreen에 진입할 때 자동 실행

### 7-4. 표시 방식

- ~~실시간 인증률 조회~~ → **폐지**
- 인증률은 **매주 주간 포트폴리오에 첨부**

  - 마감 전에 오답노트를 못 쓴 경우 → 마감 후 제출 불가

### 8-3. 다시풀기용 오답노트 (`current_error_note_text`) ✅ 확정

- 다시풀기를 완료할 때마다 **새로 작성 가능** (덮어쓰기)
- **인증률에 영향 없음**
- 마감 전·후 관계없이 current_record가 있으면 항상 작성 가능
- 시나리오 B·C 학생도 current_record에서 오답노트 작성·저장 가능

## 11. Result 파일 역할 재정의 ✅ 확정 (최후순위)

### 11-1. 현재 상태

각 Result 파일(`*-result.js`)에는 두 가지가 혼재:
- **채점 요약** (점수%, 정답수, 오답수 표시)
- **해설 상세** (지문 해석, 문항별 해설, 단어 설명)

### 11-2. V3 방향

- 채점 요약 부분 → **V3 채점 대시보드로 이관** (taskDashboardScreen 우측)
- Result 파일 = **해설 전용 뷰어**로 역할 재정의
- 이 작업은 구현 우선순위에서 **가장 마지막**에 처리

## 12. 구현 우선순위 ✅ 확정

```
1단계: 핵심 흐름
  - 스케줄 → 실전풀이 → 채점 대시보드 → 해설보기 → 메모장
  
2단계: 데이터 저장/조회
  - study_results_v3 테이블 연동
  - initial_record, current_record, 오답노트 저장
  - 라이팅 작성본 저장 (writing_email_text, writing_discussion_text)
  
3단계: 부가 기능
  - 다시풀기, 잔디, 마이페이지, 점수 추이 차트
  
4단계: 이어풀기
  - 중간 이탈 → 답안 임시저장 → 재진입 시 복원
  
5단계 (최후순위): Result 파일 분리
  - 채점 요약 코드를 대시보드로 이관
  - Result 파일을 해설 전용 뷰어로 정리
```
